# User Stories

## 用户故事 1：家教老师备课

### 用户

小学数学家教老师。

### 目标

- 快速理解手头讲义和例题
- 生成适合学生的讲解
- 围绕资料生成同类题

### 典型链路

1. 上传讲义或题单
2. 系统识别专题、年级
3. 生成教学讲解
4. 再生成几道同类型题
5. 查看来源，确认结果可信

### 价值点

- 节省备课时间
- 让资料真正变成可讲、可练的内容

## 用户故事 2：小机构老师快速出专题小测

### 用户

小机构数学老师或助教。

### 目标

- 围绕某个专题快速整理题目
- 控制题量和难度
- 导出成可直接发给学生的小测

### 典型链路

1. 上传专题讲义和例题
2. 选择资料范围
3. 生成练习题或小测
4. 检查题型和难度
5. 导出 PDF / Word

### 价值点

- 从“生成内容”升级为“生成教学交付物”

## 用户故事 3：家长围绕孩子资料做课后辅导

### 用户

小学阶段孩子的家长。

### 目标

- 看懂孩子当前在学什么
- 知道怎么给孩子讲
- 再生成几道类似题巩固

### 典型链路

1. 上传孩子当前讲义、题单、作业或拍照题
2. 系统自动识别年级、专题
3. 家长先生成讲解，看懂“应该怎么讲”
4. 再生成几道类似题
5. 导出成练习单或直接打印

### 价值点

- 帮家长降低辅导门槛
- 节省自己整理讲法和练习的时间

### 重要结论

- 家长最先要的不是题，而是“怎么讲”
- 所以默认主流程应该是：
  1. 上传资料
  2. 先生成讲解
  3. 再生成练习

---

## 当前结论

- 产品底层能力继续按老师需求做深
- 产品对外表达和商业化想象可以更偏家长
- 短期最重要的产品链路不是“问答优先”，而是“讲解优先”
- 导出 PDF / Word 是高价值增强能力，优先级高于“直连打印机”
